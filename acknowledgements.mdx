## Core inference foundation

The inference engine is based on [vLLM](https://github.com/vllm-project/vllm) with
custom patches for Grace-Blackwell (GB10) unified memory and DGX Spark topologies.
For reliable multi-node operation on fabric (QSFP / RoCE / 400G switches), we
incorporate the patched NCCL and cluster recipes from
[eugr/spark-vllm-docker](https://github.com/eugr/spark-vllm-docker).

eugr's Docker configuration and maintenance have become the community standard
for DGX Spark clustering. Many of the real-world networking and performance lessons
in AINode come directly from that work.

## What AINode adds

While the low-level clustering and inference performance build on eugr's foundation,
AINode focuses on the **full user experience**:

- Unified single-container deployment with automatic peer discovery
- Modern web UI including radial cluster topology view
- Integrated chat, model management, and OpenAI-compatible serving
- In-browser distributed training workflows (LoRA / QLoRA / fine-tuning with live metrics)
- Simplified configuration and management for both solo and clustered operation
- Prometheus metrics, version update polling, and cluster-wide update from the browser

This combination lets users go from `curl | bash` to chatting with a 4-node sharded
model — or starting a training job — without deep diving into Ray, NCCL env vars,
or custom scripts.

## If you're coming from eugr

If you already use `eugr/spark-vllm-docker` and want the next layer of usability,
AINode is designed as a natural progression. Your existing fabric config, SSH setup,
and model weights are directly compatible. AINode wraps the same inference stack in a
managed container with a browser interface.

```bash
curl -fsSL https://ainode.dev/install | bash
```

Your existing multi-node cluster topology still works — AINode's auto-discovery
finds the same nodes on the same fabric.
